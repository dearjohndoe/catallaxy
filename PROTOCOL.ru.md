# Протокол Catallaxy v1 (CTLX/1)

> [English version](PROTOCOL.md)

Статус: **стабильный, в проде** (TON mainnet). Документ описывает протокол как он реализован; заменяет черновую спеку из `PROJECT.md` (v0.2, исторический документ).

Catallaxy — permissionless-протокол продажи машинно-вызываемых сервисов и цифровых товаров за он-чейн платежи. Состоит из трёх независимых слоёв:

1. **Реестр** — он-чейн регистрация агентов через heartbeat-транзакции.
2. **Вызов** — HTTP 402 флоу «оплата → вызов» между клиентом и агентом.
3. **Репутация** — опциональные он-чейн сигналы: платежи / рефанды / оценки.

Каждый слой реализуем независимо. Соответствующий протоколу *агент* — любой HTTP-сервер, реализующий §4–§6; *клиент* — что угодно, умеющее читать реестр и платить он-чейн. Смарт-контракты не требуются.

## 1. Термины

| Термин | Значение |
|---|---|
| **Агент** | Сервис на маркетплейсе; идентифицируется парой (`sidecar_id`, адрес кошелька). Кошелёк принимает платежи и шлёт heartbeat'ы/рефанды; уникальность кошелька на агента не требуется — атрибуция между агентами на общем кошельке несётся суффиксом `sidecar_id` в нонсах и payload'ах. |
| **Сайдкар** | Референсная реализация агентской стороны (оборачивает любой stdin/stdout-скрипт). |
| **Реестр** | Публично известный адрес кошелька. Реестр ничего не подписывает — это «почтовый ящик», входящие транзакции которого читают клиенты. |
| **Рейл** | Платёжный маршрут: в v1 — `TON` (нативный) или `USDT` (TEP-74 jetton). |
| **Клиент** | Покупатель: фронтенд для людей, LLM через MCP или другой агент. |
| **SKU** | Единица товара с ценой внутри агента (у агента ≥1 SKU). |

Все протокольные payload'ы внутри транзакций — TON-селлы: 32-битный опкод, затем строка в snake-кодировке.

## 2. Опкоды

| Действие | Опкод | Тело после опкода |
|---|---|---|
| Heartbeat (регистрация/обновление) | `0xAC52AB67` | JSON, ≤ 2048 байт |
| Платёж | `0x50415900` (`PAY\0`) | строка-нонс |
| Рефанд | `0x52464E44` (`RFND`) | JSON `{"sidecar_id": ...}` |
| Оценка | `0x52617465` (`Rate`) | строка, содержащая `sidecar:{sidecar_id}` и `score:{1-5}` |

## 3. Слой реестра

**Регистрация = heartbeat.** Агент шлёт транзакцию (любая малая сумма, референс: 0.01 TON) на адрес реестра с опкодом `0xAC52AB67` и JSON-payload:

```json
{
  "sidecar_id": "…",            // обязательно, стабильная идентичность
  "endpoint": "https://…",      // обязательно, базовый URL HTTP API агента
  "name": "…", "description": "…",
  "capabilities": ["…"],
  "price": 10000000,             // наноTON, ориентировочная (см. §5)
  "price_usdt": 1000000,         // микро-USDT, опционально
  "args_schema": { … },          // JSON-схема body для /invoke
  "result_schema": { … },        // опционально
  "has_quote": true,             // опционально: агент поддерживает POST /quote
  "owner_wallet": "…",          // опционально
  "preview_url": "…", "avatar_url": "…", "images": ["…"]  // опциональные медиа
}
```

**Окно живости: 7 дней.** Клиент считает агента залистенным ⟺ есть heartbeat за последние 7 дней. Обновление = новый heartbeat (побеждает свежий). Дерегистрация = молчание.

**Правило discovery (клиентская сторона):** забрать входящие транзакции реестра за 7 дней, отфильтровать по опкоду `0xAC52AB67`, распарсить JSON, выбросить записи без `endpoint` или `sidecar_id`, дедуплицировать по `sidecar_id`, оставив новейшую.

Адрес реестра — дефолт клиентской конфигурации, а не константа протокола: любой может поднять альтернативный реестр, указав клиентам другой адрес.

## 4. HTTP API агента

| Роут | Назначение |
|---|---|
| `POST /invoke` | Прифлайт (без оплаты) → 402; оплаченный вызов (с пруфом) → результат |
| `POST /quote` | Опционально: залочить цену/сток, возвращает `quote_id` |
| `GET /result/{job_id}` | Поллинг асинхронной задачи |
| `GET /download/{file_id}` | Скачать файл, созданный задачей |
| `GET /info` | Живые метаданные: имя, capabilities, `args_schema`, `payment_rails`, `skus[]` с текущими ценами/стоком |

`/info` — авторитетный источник цен и стока; значения из heartbeat — ориентировочные подсказки для листинга.

## 5. Флоу вызова (HTTP 402)

```
Клиент                                Агент
  │ POST /invoke {capability, body,     │
  │   rail, sku?}                       │
  │────────────────────────────────────►│
  │ 402 {payment_request,               │
  │      payment_options[]}             │
  │◄────────────────────────────────────│
  │                                     │
  │   он-чейн платёж (§6)               │
  │                                     │
  │ POST /invoke {tx, nonce, rail,      │
  │   capability, body, sku?}           │
  │────────────────────────────────────►│
  │ 200 {status:"done", result}         │
  │   или {job_id, status:"pending"}    │
  │◄────────────────────────────────────│
  │ GET /result/{job_id}  (если pending)│
```

**Тело 402-ответа:**

```json
{
  "error": "Payment required",
  "payment_request": { … },        // первый из payment_options, legacy
  "payment_options": [
    { "rail": "TON",  "address": "EQ…", "amount": "10000000",
      "memo": "<nonce>", "sku": "default" },
    { "rail": "USDT", "address": "EQ…", "amount": "1000000",
      "memo": "<nonce>", "sku": "default",
      "token": {"symbol": "USDT", "master": "EQ…", "decimals": 6} }
  ]
}
```

Заголовки `x-ton-pay-address`, `x-ton-pay-amount`, `x-ton-pay-nonce` дублируют TON-опцию для клиентов, читающих только хедеры.

**Нонс:** `{16 hex-символов}:{sidecar_id}`. Суффикс привязывает платёж к одному агенту: клиент НЕ ДОЛЖЕН платить по нонсу с чужим суффиксом; агент ДОЛЖЕН отклонять чужие нонсы.

**Семантика ошибок:**

| Статус | Значение |
|---|---|
| `402` | Нет/невалиден пруф оплаты; в теле — платёжные опции |
| `409 {"error":"out_of_stock"}` | SKU недоступен к покупке (в т.ч. когда динамическое ценообразование не вернуло цену) |
| `409` (tx already processed) | Повтор уже использованного `tx` |
| `503` + `Retry-After` | Агент сейчас не может верифицировать платежи (монитор чейна деградировал). Агент ДОЛЖЕН отказаться выдавать 402, а не принимать неверифицируемые платежи |

## 6. Платёжные транзакции

**TON-рейл.** Перевод на `address` суммы ≥ `amount` наноTON, тело сообщения — селл: `0x50415900` (32 бита) ++ нонс snake-строкой. Нонс едет в payload, **не** в текстовом комментарии.

**USDT-рейл.** Стандартный TEP-74 jetton-перевод (опкод `0x0F8A7EA5`) суммы ≥ `amount` микро-USDT, отправленный на *собственный* jetton-кошелёк плательщика, с тем же платёжным селлом в `forward_payload` и ~0.07 TON на газ. Агент наблюдает результирующий `transfer_notification` (`0x7362D09C`) на своём jetton-кошельке.

**Правила верификации (сторона агента).** Пруф `{tx, nonce}` валиден ⟺ выполняется всё:
1. транзакция существует и финализирована он-чейн;
2. получатель — кошелёк агента (TON) или USDT jetton-кошелёк агента (USDT);
3. переведённая сумма ≥ котированного `amount` для выбранного рейла и SKU;
4. body/forward_payload несёт опкод `0x50415900` и нонс, равный выданному, с суффиксом собственного `sidecar_id` агента;
5. платёж моложе платёжного таймаута агента;
6. `tx` никогда ранее не был использован (одноразовость; агент ДОЛЖЕН долговременно хранить использованные tx-хэши).

## 7. Рефанды

Если верифицированный платёж невозможно обслужить (сбой агента, проигранная гонка за сток, неподдерживаемый рейл), агент СЛЕДУЕТ вернуть деньги плательщику со своего кошелька: сумма платежа минус фиксированная комиссия, опкод `0x52464E44`, JSON-тело `{"sidecar_id": …}` — чтобы рефанд был атрибутируем он-чейн. Рефанды — best-effort, с ретраями и бэкоффом; в v1 **нет эскроу** — см. §9.

## 8. Слой репутации (опционально)

Любой кошелёк МОЖЕТ отправить оценку на кошелёк агента: опкод `0x52617465`, тело содержит `sidecar:{sidecar_id}` и `score:{1-5}`. Клиенты считают репутацию по он-чейн истории; референсные веса: платёж `+2.0`, рефанд `−2.1`, оценка `(score−3)×1.25`, нормировка в 0–10. Веса — политика клиента, не протокол.

## 9. Соображения безопасности

- **Модель доверия с предоплатой.** Клиент платит до доставки; рефанды добровольны. v1 экономически подходит для дешёвых вызовов, где риск ценится репутацией, а не эскроу. Эскроу — кандидат в v2.
- **Replay.** Одноразовые tx-хэши (§6.6) и нонсы, привязанные к агенту (§5), исключают повтор между вызовами и между агентами.
- **Sybil-оценки.** Входы репутации permissionless; клиентам СЛЕДУЕТ дисконтировать паттерны самонакрутки (платежи с кошельков, аффилированных с агентом).
- **Подмена 402 через прокси.** Если агент доступен через reverse proxy, прокси может переписать `payment_options`. Клиентам СЛЕДУЕТ сверять `address` с `owner_wallet` из heartbeat, когда тот есть; подпись ответа запланирована в v2.
- **Спам реестра.** Листинг permissionless; клиент ДОЛЖЕН считать содержимое heartbeat недоверенным вводом (лимиты размера схем, санитизация URL).

## 10. Версионирование

Это CTLX/1, по построению TON-only (селлы, опкоды, jetton'ы). CTLX/2 (в проектировании) обобщает рейл до пары *(chain, asset)*, выносит схемы из heartbeat в `/info` (тонкий он-чейн payload, ≤ ~900 байт), вводит chain-agnostic пруф платежа `{rail, proof, nonce}` и добавляет x402-совместимый режим.
